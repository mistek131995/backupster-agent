## Бэкап

Физический бэкап снимает весь кластер PostgreSQL инструментом `pg_basebackup`. Он подключается к серверу по стандартному сетевому протоколу репликации — никакой общей папки не нужно. Агент может находиться в отдельном контейнере, на другом хосте или внутри Docker Compose — `pg_basebackup` работает через TCP так же, как `pg_dump`.

На выходе получается один файл `base.tar.gz`: архив PGDATA со встроенным WAL. Файл проходит стандартный pipeline: шифрование AES-256-GCM → загрузка в S3/SFTP.

Бинарник `pg_basebackup` должен быть доступен на хосте агента (пакет `postgresql-client`). Версия клиента должна быть ≥ версии сервера.

## Восстановление

Здесь ситуация принципиально иная.

`pg_basebackup` для бэкапа работает по сети. Восстановление — нет: агент должен **физически заменить содержимое PGDATA** и **перезапустить кластер** через `pg_ctl`. Оба действия невозможны по сети — нужен прямой доступ к файловой системе и к процессу PostgreSQL.

Агент последовательно:

1. Скачивает и расшифровывает `base.tar.gz`.
2. Запрашивает у живого PostgreSQL путь к PGDATA (`SHOW data_directory`), проверяет, что путь виден с хоста агента.
3. Останавливает кластер: `pg_ctl stop -m fast -w`.
4. Очищает PGDATA.
5. Распаковывает архив в PGDATA: `tar -xzf base.tar.gz -C <pgdata>`.
6. Запускает кластер: `pg_ctl start -w`.

Из этого следует одно жёсткое требование: **агент должен выполняться на том же хосте, что и PostgreSQL**, и иметь доступ к его файловой системе.

## Сценарии развёртывания

| Сценарий | Бэкап | Восстановление |
|---|---|---|
| Агент и PG на одном хосте (без Docker) | ✅ | ✅ |
| Агент и PG в одном контейнере | ✅ | ✅ |
| Docker Compose: общий volume PGDATA у обоих контейнеров + пробросить UID агента совпадает с UID postgres | ✅ | ✅ (см. ниже) |
| Агент в Docker, PG на хосте; PGDATA смонтирован в контейнер агента | ✅ | ✅ (см. ниже) |
| Агент и PG в разных контейнерах без общего PGDATA volume | ✅ | ❌ |
| Агент и PG на разных хостах | ✅ | ❌ |

### Docker Compose с общим volume

Для восстановления нужно:

1. Пробросить PGDATA-volume в контейнер агента. Путь в агенте должен совпадать с тем, что возвращает `SHOW data_directory` на PG-сервере (или быть тем же физическим томом — тогда пути одинаковые).
2. Пользователь, от которого работает агент, должен иметь права на запись в PGDATA (`chown` или совпадение UID).
3. `pg_ctl` должен быть доступен в контейнере агента.
4. Чтобы `pg_ctl stop/start` работал для PG в **другом** контейнере, нужно либо пробросить Docker-сокет (`/var/run/docker.sock`) и управлять контейнером через `docker stop/start`, либо запускать агент и PG в одном контейнере.

На практике Docker Compose с физическим restore — нестандартная конфигурация. Для containerized-сред проще использовать логический бэкап. Физический режим обычно применяется на bare-metal или VM.

## Требуемые права

### Бэкап

Пользователь в `ConnectionConfig` должен иметь привилегию `REPLICATION` или быть superuser:

```sql
CREATE ROLE backup_user WITH LOGIN PASSWORD '...' REPLICATION;
```

Также в `pg_hba.conf` нужна запись для replication-подключения с хоста агента:

```
host  replication  backup_user  <agent-ip>/32  scram-sha-256
```

### Восстановление

Для выдачи команды `SHOW data_directory` достаточно права на подключение. Всё остальное — доступ к PGDATA и `pg_ctl` — определяется на уровне ОС, не PostgreSQL.

Агент проверяет до начала разрушительных операций:
- `pg_ctl` есть в PATH;
- PGDATA, полученный из `SHOW data_directory`, доступен с хоста агента.

Если хотя бы одно условие не выполнено — задача завершается ошибкой до остановки кластера.

## Когда не подходит физический режим

Физический бэкап — **снимок всего кластера**, а не отдельной базы. Если нужно:

- восстановить одну базу из кластера с несколькими базами;
- перенести базу на другой сервер с другой версией PG (мажорный апгрейд);
- получить бэкап, который читается как обычный SQL;

— используйте логический режим (`pg_dump`). Логический бэкап агент может снимать и восстанавливать через сеть без каких-либо требований к совместному расположению.

Физический режим подходит, когда важна скорость снятия/восстановления на больших базах и агент гарантированно co-located с PostgreSQL.
