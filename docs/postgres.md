## Логический бэкап

Логический бэкап снимает **одну базу данных** инструментом `pg_dump` и сохраняет её в виде SQL-скрипта. Агент может находиться в любом месте сети — доступ к файловой системе PostgreSQL не нужен ни для бэкапа, ни для восстановления.

### Бэкап

`pg_dump` подключается к серверу по стандартному TCP-соединению и выгружает схему и данные выбранной базы в формате plain SQL (`-F p`). Агент стримит вывод через pipeline: сжатие gzip → запись в `OutputPath` на диске. Полученный `.sql.gz` файл затем проходит шифрование AES-256-GCM → загрузка в S3/SFTP.

Дамп генерируется с флагами `--clean --if-exists`: перед каждым объектом идут `DROP … IF EXISTS`. При восстановлении это безвредно — база пересоздаётся заново перед разворачиванием дампа.

**Требования**

- Бинарник `pg_dump` доступен на хосте агента (пакет `postgresql-client`). Версия клиента ≥ версии сервера.
- Пользователь в `ConnectionConfig` имеет права на подключение к базе и чтение всех схем/таблиц, которые нужно сохранить. Минимум:

  ```sql
  CREATE ROLE backup_user WITH LOGIN PASSWORD '...';
  GRANT CONNECT ON DATABASE mydb TO backup_user;
  GRANT USAGE ON SCHEMA public TO backup_user;
  GRANT SELECT ON ALL TABLES IN SCHEMA public TO backup_user;
  -- для будущих таблиц:
  ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT TO backup_user;
  ```

- Стандартная запись в `pg_hba.conf` для обычного подключения (как для приложения). Replication-привилегии не нужны.

### Восстановление

Восстановление выполняется через `psql`. Кластер PostgreSQL не останавливается, остальные базы на сервере не затрагиваются.

Агент последовательно:

1. Скачивает и расшифровывает дамп в временный файл на диске.
2. Отключает все активные соединения к целевой базе: `pg_terminate_backend`.
3. Удаляет целевую базу (если существует): `DROP DATABASE IF EXISTS <target>`.
4. Создаёт её заново: `CREATE DATABASE <target>`.
5. Разворачивает дамп: `psql -f <файл> -v ON_ERROR_STOP=1` по сети — при первой SQL-ошибке restore немедленно прерывается.

**Требования**

- Бинарник `psql` доступен на хосте агента (пакет `postgresql-client`).
- Пользователь в `ConnectionConfig` имеет привилегию `CREATEDB` и роль `pg_signal_backend` (для отключения активных соединений перед drop), либо является superuser:

  ```sql
  ALTER ROLE backup_user WITH CREATEDB;
  GRANT pg_signal_backend TO backup_user;
  ```

- Агент может находиться где угодно — только TCP-доступ к серверу.

**Что нужно знать**

- Целевая база **всегда пересоздаётся**: все существующие данные в ней уничтожаются. Активные соединения к ней принудительно отключаются перед drop.
- Можно восстановить в базу с **другим именем** или на **другой сервер** — в том числе с другой мажорной версией PostgreSQL (pg_dump/psql совместимы между версиями).
- Восстановление медленнее физического на больших базах: данные вставляются построчно через SQL, индексы перестраиваются.
- Во время restore целевая база недоступна (дропнута и пересоздаётся). Остальные базы на кластере не затрагиваются.

## Физический бэкап

Физический бэкап снимает весь кластер PostgreSQL целиком на уровне файлов PGDATA. Инструмент — `pg_basebackup`. На выходе — файл `{database}_{timestamp}.tar.gz` (архив PGDATA со встроенным WAL), который проходит стандартный pipeline: шифрование AES-256-GCM → загрузка в S3/SFTP.

> **Tablespaces не поддерживаются.** Если в кластере есть пользовательские tablespaces (любые, кроме `pg_default` и `pg_global`), физический бэкап **отвергается на preflight'е** с понятной ошибкой. Причина: `pg_basebackup --format=tar` пишет данные каждого tablespace в отдельный `<oid>.tar`, а Backupster шиппит только `base.tar.gz`. Без guard'а получился бы тихо неполный бэкап и битый кластер на restore (симлинки в `pg_tblspc/` указывали бы в никуда). Если в кластере есть tablespaces — используйте логический режим, он от tablespaces не зависит.

### Бэкап

`pg_basebackup` подключается к серверу по стандартному протоколу репликации. Перед запуском агент выполняет preflight-проверку: убеждается, что `pg_basebackup` и `pg_ctl` есть в PATH, затем выполняет `SHOW data_directory` и проверяет, что полученный путь PGDATA доступен на хосте агента. Если хотя бы одно условие не выполнено — бэкап завершается ошибкой без каких-либо операций с данными.

**Требования**

- Бинарники `pg_basebackup` и `pg_ctl` доступны на хосте агента (пакет `postgresql`, не только `postgresql-client`). Версия ≥ версии сервера.
- PGDATA, возвращаемый `SHOW data_directory`, доступен на хосте агента. Это означает, что агент должен выполняться на том же хосте, что и PostgreSQL, или PGDATA-volume смонтирован в контейнер агента с совпадающим путём.
- Пользователь в `ConnectionConfig` имеет привилегию `REPLICATION` или является superuser:

  ```sql
  CREATE ROLE backup_user WITH LOGIN PASSWORD '...' REPLICATION;
  ```

- В `pg_hba.conf` есть две записи с хоста агента — обычное подключение (для preflight `SHOW data_directory`) и replication (для `pg_basebackup`):

  ```
  host  postgres     backup_user  <agent-ip>/32  scram-sha-256
  host  replication  backup_user  <agent-ip>/32  scram-sha-256
  ```

**Поиск бинарников `pg_*`.** Агент сам определяет, откуда звать `pg_dump`, `pg_basebackup`, `psql`, `pg_ctl`, ориентируясь на **мажорную версию сервера**, с которым поднимает соединение. Алгоритм такой:

1. Если в `ConnectionConfig.BinPath` указан путь — используется он.
2. Иначе агент спрашивает у сервера `SHOW server_version_num` и ищет соответствующий каталог установки:
   - Windows: реестровый ключ `HKLM\SOFTWARE\PostgreSQL\Installations\postgresql-x64-<MAJOR>` (`Base Directory`), затем `C:\Program Files\PostgreSQL\<MAJOR>\bin`.
   - Debian/Ubuntu: `/usr/lib/postgresql/<MAJOR>/bin`.
   - RHEL/PGDG: `/usr/pgsql-<MAJOR>/bin`.
3. Если ни один каталог не найден — fallback на `PATH`.

На хосте с несколькими мажорными версиями PostgreSQL агент автоматически выберет клиент той же версии, что и сервер — именно это нужно, потому что `pg_basebackup` отказывается работать с сервером другой мажорной версии (ошибка `incompatible server version`). `PATH` в такой конфигурации становится хрупким: `PATH`-кеш `services.exe` обновляется только при перезагрузке Windows, порядок установок меняет приоритет, и так далее.

**Override в конфиге.** Если установка нестандартная или авто-резолв не подходит, задайте каталог явно в `Connections[]`:

```json
{
  "Connections": [
    {
      "Name": "pg-main",
      "DatabaseType": "Postgres",
      "Host": "127.0.0.1",
      "Port": 5432,
      "Username": "backup_user",
      "Password": "...",
      "BinPath": "C:\\Program Files\\PostgreSQL\\18\\bin"
    }
  ]
}
```

Override бьёт всё остальное — ни сервер, ни реестр, ни `PATH` не опрашиваются.

### Восстановление

Восстановление принципиально отличается от бэкапа. Агент должен **физически заменить содержимое PGDATA** и **перезапустить кластер** через `pg_ctl`. Оба действия невозможны по сети — нужен прямой доступ к файловой системе и к процессу PostgreSQL.

Агент последовательно:

1. Резолвит `pg_ctl` — по тому же правилу, что и `pg_basebackup` (см. раздел про поиск бинарников выше): override `BinPath` → реестр/стандартный каталог для мажорной версии сервера → `PATH`.
2. Запрашивает у живого PostgreSQL путь к PGDATA (`SHOW data_directory`), проверяет, что путь виден с хоста агента, и убеждается, что postmaster **не управляется** systemd-юнитом или Windows-сервисом (см. ниже про **service-manager guard**).
3. Скачивает и расшифровывает `base.tar.gz`.
4. Распаковывает архив в staging-каталог `<pgdata>.new-<id>` рядом с PGDATA. Живой кластер продолжает работать.
5. Проверяет staging: есть `PG_VERSION` и `global/`, мажорная версия в `PG_VERSION` совпадает с `pg_ctl --version`. Если на шагах 1–5 что-то пошло не так — staging удаляется, живой кластер не трогается, задача завершается с ошибкой.
6. Останавливает кластер: `pg_ctl stop -D <pgdata> -m fast -w`.
7. Атомарный swap: `<pgdata>` → `<pgdata>.old-<id>`, затем `<pgdata>.new-<id>` → `<pgdata>`. Оба переименования — в одном родительском каталоге, на стандартных ФС это atomic rename.
8. Запускает кластер: `pg_ctl start -D <pgdata> -w -t <RestoreSettings.PgCtlStartTimeoutSeconds>` (дефолт 600 секунд). Если старт упал или не уложился в таймаут — агент сначала делает best-effort `pg_ctl stop -m immediate` (на случай, если postmaster уже детачнулся и работает сам по себе, держа файлы и порт), затем откатывает: `<pgdata>` → `<pgdata>.failed-<id>` (оставляется на диске для диагностики, автоматически не удаляется), `<pgdata>.old-<id>` возвращается на место как `<pgdata>`. Старый кластер остаётся остановленным — оператор решает, поднимать его заново или нет.
9. После успешного старта `<pgdata>.old-<id>` удаляется.

**Требования**

- Агент выполняется на том же хосте, что и PostgreSQL, или PGDATA-volume смонтирован в контейнер агента с совпадающим путём.
- Пользователь, от которого работает агент, имеет права на запись в PGDATA **и в родительский каталог PGDATA** (atomic swap использует `rename` на родителе).
- На той же файловой системе, что и PGDATA, есть свободное место под распакованную копию кластера — на пике restore держит **старый PGDATA + новый PGDATA** (≈ 2× размер живого кластера). Staging и `.old-<id>` располагаются рядом с PGDATA, а не в `TempPath`, чтобы swap был атомарным.
- Для больших кластеров с долгим WAL-replay **поднимите `RestoreSettings.PgCtlStartTimeoutSeconds`** (дефолт 600 секунд). pg_ctl сдаётся по таймауту, а postmaster к тому моменту уже детачнут — агент его потом добивает через `pg_ctl stop -m immediate` перед откатом, но лучше до этого не доводить.
- `pg_ctl` доступен агенту — автоматически резолвится под мажорную версию сервера либо задан через `ConnectionConfig.BinPath`, либо лежит в `PATH`.
- Для управления PG в **другом** контейнере нужно либо пробросить Docker-сокет (`/var/run/docker.sock`) и управлять контейнером через `docker stop/start`, либо запускать агент и PG в одном контейнере.

Все проверки и распаковка (шаги 1–5) выполняются до остановки кластера. Любая ошибка на этих шагах оставляет живое PGDATA нетронутым — отменить физическое восстановление можно вплоть до `pg_ctl stop`.

**Сценарии развёртывания**

| Сценарий | Бэкап | Восстановление |
|---|---|---|
| Агент и PG на одном хосте (bare-metal / VM) | ✅ | ✅ |
| Агент и PG в одном контейнере | ✅ | ✅ |
| Docker Compose: общий PGDATA-volume, совпадающий UID | ✅ | ✅ |
| Агент в Docker, PG на хосте; PGDATA смонтирован в контейнер | ✅ | ✅ |
| Агент и PG в разных контейнерах без общего PGDATA-volume | ❌ | ❌ |
| Агент и PG на разных хостах | ❌ | ❌ |

### Трудности и ограничения

Физический бэкап — мощный инструмент, но он предъявляет нетривиальные требования к инфраструктуре. Ниже — полный перечень сложностей, которые нужно учесть **до** выбора физического режима.

**Восстановление требует простоя и двойного места на диске.**
На время swap кластер остановлен, все базы на нём недоступны. Пиковое использование диска ≈ 2× размера живого PGDATA: staging-каталог распакованного бэкапа и старая копия (`.old-<id>`) сосуществуют рядом с PGDATA до успешного старта. После успеха `.old-<id>` удаляется автоматически.

**Swap атомарный, с автоматическим откатом.**
Архив распаковывается не поверх PGDATA, а в staging-каталог рядом. Только после проверки структуры staging и остановки кластера выполняется пара переименований. Если новый кластер не стартует, агент возвращает старый PGDATA на место, а несработавшую копию оставляет как `.failed-<id>` для диагностики (вручную не удаляется). До `pg_ctl stop` живое PGDATA не трогается совсем — отменить физическое восстановление можно в любой момент.

**Снимок всего кластера, а не отдельной базы.**
Нельзя восстановить одну базу из многобазового кластера — восстанавливается всё целиком. Если на кластере живут производственные базы нескольких сервисов, restore одной из них ронит остальные. **Перед physical restore проверьте, какие БД сейчас живут в target-кластере** (`psql -l` или `SELECT datname FROM pg_database WHERE datistemplate = false;`) — все, кроме `template0`/`template1`, будут заменены данными из бэкапа. Backupster не перечисляет их автоматически в UI: решение «стереть всё» — ваше.

Связанный guard на бэкенде: для PG physical API-эндпоинт `POST /api/v1/agents/{id}/restore` **отклоняет непустой `targetDatabaseName`** (400), потому что physical не умеет rename. Через UI поле «target-имя БД» для physical-бэкапов и так скрыто; guard закрывает только прямые вызовы API мимо UI.

**Агент должен быть co-located с PostgreSQL — и для бэкапа, и для restore.**
Агент проверяет доступность PGDATA ещё до запуска `pg_basebackup`: если путь недоступен, бэкап не стартует. Это значит, что удалённый агент (другой хост, отдельный контейнер без смонтированного volume) не сможет ни снять бэкап, ни восстановить его. В containerized-средах (Kubernetes, Docker без shared volume) физический режим практически нереализуем без специальной конфигурации.

**Бинарный формат, привязан к версии.**
`base.tar.gz` содержит бинарные файлы PGDATA. Восстановить на PostgreSQL другой мажорной версии нельзя — сервер откажется стартовать. Для миграции между версиями (например, PG 15 → PG 16) физический бэкап не подходит.

**REPLICATION-привилегия и правка pg_hba.conf.**
Стандартный пользователь с `SELECT`-правами для бэкапа не годится. Нужно выдать `REPLICATION`-роль и явно разрешить replication-подключение в `pg_hba.conf`. На управляемых PostgreSQL (RDS, Cloud SQL, Supabase) эта привилегия часто недоступна или ограничена.

**Tablespaces не поддерживаются.**
Если в кластере есть пользовательские tablespaces (всё, кроме `pg_default` и `pg_global`) — physical-режим падает на preflight'е бэкапа и на sanity-проверке restore (последнее ловит legacy-архивы, созданные до guard'а). Их данные физически живут вне PGDATA, в отдельных каталогах, на которые `pg_tblspc/<oid>` ссылается симлинками. `pg_basebackup` упаковывает их в отдельные tar-файлы, которые Backupster не шиппит — без guard'а вылез бы тихо битый бэкап. Кластерам с tablespaces — только logical.

**Кластер не должен быть под управлением service manager'а (только для restore).**
Перед swap'ом restore-провайдер читает PID из `postmaster.pid` и проверяет:
- Linux: `/proc/<pid>/cgroup` — если PID живёт в `system.slice/<имя>.service`, кластером управляет systemd;
- Windows: PowerShell `Get-CimInstance Win32_Service -Filter 'ProcessId=<pid>'` — если процесс хостится сервисом, поле `Name` будет непустым.

Если детектор сработал — задача отвергается до любых действий с диском, кластер продолжает работать. Сообщение содержит конкретное имя юнита/сервиса и команды для отключения автозапуска (`systemctl disable …` / `Set-Service … -StartupType Manual`). Причина: между нашим `pg_ctl stop` и `Directory.Move` сервис-менеджер с `Restart=always` (или Recovery → Restart на Windows) может попытаться перезапустить postmaster, гонка ловит swap в самый неудачный момент. Плюс после успешного restore наш `pg_ctl start`-нутый postmaster живёт **вне** управления сервиса — `systemctl status` врёт, а попытка `systemctl start` поднимает второй экземпляр и валится.

Сценарий «агент в контейнере, PG на хосте как systemd-юнит» детектор обойти не сможет (PID хоста не виден в `/proc` контейнера) — fallthrough в этом случае безопасен, потому что `pg_ctl stop` всё равно не достанется до postmaster в чужом PID-namespace и упадёт раньше с ошибкой от самого pg_ctl.

**pg_ctl должен быть доступен агенту.**
Агент вызывает `pg_ctl` напрямую. По умолчанию он находит его авто-резолвом под мажорную версию сервера (реестр / стандартные каталоги / `PATH`). В минимальных Docker-образах (`postgres:alpine`) `pg_ctl` есть, в образах только с клиентскими утилитами (`postgresql-client`) — нет. Версия должна совпадать с версией сервера; если нужно — путь задаётся явно в `ConnectionConfig.BinPath`.

**Когда физический режим оправдан.**
Физический бэкап подходит в первую очередь для bare-metal и VM, где агент co-located с PostgreSQL, база большая (десятки гигабайт и выше) и скорость снятия/восстановления критична. В containerized-средах проще использовать логический бэкап.

## Дифференциальный бэкап

Дифференциальный физический бэкап снимает только изменения с последнего полного. Подходит, когда полный бэкап тяжёлый по времени/трафику, а данные меняются понемногу: вместо полного снимка каждые 6 часов — один полный в сутки и несколько дифференциальных между ними.

В терминах Backupster дифференциальный режим — это отдельный `BackupMode = physicalDifferential` (наравне с `physical` и `logical`), со своим расписанием. Каждая дифференциальная запись ссылается на корневой полный через поле `BaseBackupRecordId` — цепочка плоская, не «лесенкой»: и первый, и десятый DIFF указывают на один и тот же FULL, а не на предшествующий DIFF.

### Как работает

Дифференциал реализован поверх нативной механики PostgreSQL 17+: `pg_basebackup --incremental=<backup_manifest>`. Сервер сравнивает текущий PGDATA с манифестом предыдущего бэкапа (по WAL-сумаризации) и пишет в архив только изменившиеся куски страниц.

Процесс на агенте:

1. **Резолв родителя.** По cron-расписанию `physicalDifferential` агент сначала дёргает `GET /api/v1/agent/backup-records/last-successful?database=…&storage=…&mode=physical` (см. [NETWORK.md](../NETWORK.md), секция 9.5). Дашборд возвращает ID последнего успешного полного бэкапа для этой пары `(БД, хранилище)`. Если родителя нет — DIFF на этом тике пропускается с предупреждением в лог; никаких автоматических «сделаю FULL вместо DIFF» нет.
2. **Открытие записи.** `POST /api/v1/agent/backup-record` с `backupMode=physicalDifferential` и `baseBackupRecordId=<id из шага 1>`. Дашборд валидирует цепочку (родитель `Success`, тот же агент/БД/хранилище, не помечен `StorageUnreachable`) и возвращает в ответе ключи родителя — `baseDumpObjectKey` и `basePgBaseManifestKey`.
3. **Подтягивание манифеста.** Агент скачивает `backup_manifest.enc` родителя из хранилища, расшифровывает в `OutputPath`, отдаёт путь в `pg_basebackup --incremental=<path>`.
4. **Снятие архива.** `pg_basebackup --format=tar --gzip --incremental=<path>` пишет дифференциальный архив `{database}_{yyyyMMdd_HHmmss}_diff.tar.gz` и свежий `backup_manifest` рядом. Оба файла шифруются и загружаются — манифест нужен следующим DIFF'ам и `pg_combinebackup` на restore.

Полный физический бэкап (FULL) теперь тоже всегда сохраняет `backup_manifest` рядом с дампом — без него по нему нельзя будет построить ни один последующий дифференциальный. Это лёгкий файл (десятки/сотни КБ), переплата по месту незаметная.

### Восстановление цепочки

Восстановить можно любую точку цепочки — пользователь выбирает конкретный дифференциал в UI. Дашборд раскрывает цепочку (FULL + все интермедиэйт-дифференциалы ≤ выбранной точки по `BackupAt`) и кладёт упорядоченный массив в `RestoreTaskPayload.chain` (см. [NETWORK.md](../NETWORK.md), секция 9).

Агент:

1. Скачивает и расшифровывает каждое звено цепочки (`.tar.gz` + `backup_manifest`) во временные подкаталоги.
2. Распаковывает каждый `.tar.gz` в свой каталог `chain-N/`.
3. Запускает `pg_combinebackup -o <staging> chain-0 chain-1 … chain-N` — нативная утилита PostgreSQL 17+ склеивает FULL + все промежуточные DIFF в единый PGDATA-каталог.
4. Дальше — обычный physical-restore: атомарный swap, перезапуск кластера через `pg_ctl` (вся swap-логика та же, что у обычного physical-restore, переиспользована).

Восстановление под другим именем БД для `physicalDifferential`, как и для `physical`, не поддерживается — physical restore меняет кластер целиком. API отвергает непустой `targetDatabaseName` с 400.

### Требования и ограничения

- **PostgreSQL 17 или новее.** До PG 17 `pg_basebackup --incremental` не существует. Агент сам проверяет версию через `SHOW server_version_num` и отказывается работать на старых версиях с понятной RU-ошибкой.
- **`summarize_wal = on` на сервере PostgreSQL.** Это требование `pg_basebackup --incremental`: без активного WAL-summarizer'a сервер не знает, какие куски страниц изменились. Включается одной командой, перезапуск сервера не нужен:

  ```sql
  ALTER SYSTEM SET summarize_wal = 'on';
  SELECT pg_reload_conf();
  ```

  По умолчанию в PG 17 — `off`. Без этой настройки дифференциальный бэкап будет падать с ошибкой от самого `pg_basebackup`.
- **`pg_combinebackup` в PATH или `BinPath`.** Утилита приходит вместе с пакетом `postgresql` 17+ — там же, где `pg_basebackup` и `pg_ctl`. Резолвится тем же механизмом version-aware lookup (`SHOW server_version_num` → стандартные каталоги Windows-реестра / Debian / RHEL → `PATH` или `BinPath`).
- **Без живого FULL дифференциалы не запускаются.** Если для пары `(БД, хранилище)` нет ни одного успешного `Physical`-бэкапа — `physicalDifferential`-расписание молча пропускается с предупреждением в лог. В UI расписание подсвечивается как «нет полного — дифференциальные не запускаются».
- **Цепочка живёт в одном хранилище.** Cross-storage цепочки не поддерживаются. Если у БД сменилось `StorageName` — на новом хранилище нужно сначала снять новый FULL, иначе дифференциалы туда не пойдут.
- **Retention защищает FULL с живыми DIFF.** Полный бэкап нельзя удалить (ни вручную, ни автоматически по retention), пока на нём висит хоть один живой Successful дифференциальный. Сначала истекают и удаляются дифференциалы (листья), потом — корень.
- **Tablespaces по-прежнему не поддерживаются** — ограничение унаследовано от обычного physical, см. выше.
