# Сборщик мусора и очистка устаревших бэкапов

Два фоновых работника, которые вместе дают полную очистку хранилища:

- **Сборщик мусора для чанков** — подчищает осиротевшие куски в общем пуле `chunks/` файлового бэкапа.
- **Очистка устаревших бэкапов** — удаляет дампы и манифесты, у которых истёк срок хранения по тарифу.

Связка: retention удаляет дамп+манифест → следующий тик GC видит, что чанки больше никем не упомянуты, и подбирает их после grace-окна.

## Сборщик мусора для чанков (S3)

Файловый бэкап дедуплицирует куски в общий пул `chunks/{sha256}` — один и тот же чанк, встретившийся в нескольких бэкапах, лежит в хранилище в одном экземпляре. Когда какой-то бэкап (манифест) удаляют, его чанки могут оказаться осиротевшими, если они не используются ни одним другим бэкапом.

Агент периодически их находит и удаляет:

1. По таймеру (по умолчанию раз в 24 часа) сканирует каждое S3-хранилище.
2. Листит все манифесты (оба суффикса: `*/manifest.json.gz.enc` новых бэкапов и `*/manifest.json.enc` легаси), стримит из каждого `FileEntry` и собирает sha256 всех упомянутых чанков в одно множество. Новый формат читается через `PipeReader` без разворачивания в RAM, легаси — десериализуется целиком.
3. Листит `chunks/` и удаляет каждый объект, которого нет в множестве **и** который старше grace-окна (по умолчанию 24 часа — защита от гонки, когда параллельный бэкап уже залил чанк, но ещё не обновил манифест).

Особенности:

- **S3, Azure Blob и LocalFs.** У SFTP и WebDAV файлового бэкапа и чанков нет, удалять нечего — sweep пропускает такие хранилища с `Debug` в логе. На LocalFs sweep делает то же самое, что на S3 и Azure: рекурсивный листинг через `Directory.EnumerateFiles` (`chunks/` + манифесты), затем `File.Delete` для осиротевших.
- **Первый запуск** — через 10 минут после старта агента, чтобы не конкурировать с начальной синхронизацией.
- **Не пересекается с backup/restore.** На время sweep агент берёт общий lock — если идёт бэкап или восстановление, GC ждёт.
- **Дамп и manifest GC не трогает** — подчищается только пул `chunks/`. Удаление дамп-папок по retention — см. ниже.
- **Ошибка при чтении/дешифровке любого манифеста прерывает sweep этого хранилища.** Иначе можно удалить чанки, которые на самом деле ещё нужны.

## Настройка `GcSettings`

```json
"GcSettings": {
  "Enabled": true,
  "IntervalHours": 24,
  "GraceHours": 24
}
```

Всё опционально — блок в шаблон `appsettings.json` не попадает, по умолчанию GC включён с параметрами 24/24.

- **`Enabled`** — выключает worker целиком. `false` если хотите управлять очисткой вручную или делать её внешним инструментом.
- **`IntervalHours`** — как часто запускать sweep. Минимум 1 час (меньшие значения игнорируются). Для больших хранилищ с десятками тысяч объектов разумно поднять до 48–168 — каждый sweep читает и дешифрует все манифесты.
- **`GraceHours`** — не трогать чанки, созданные в последние N часов. Защита от гонки: если параллельный бэкап уже залил чанк, но ещё не записал манифест, чанк временно выглядит осиротевшим. Минимум 0, но ставить ниже длительности самого долгого бэкапа опасно.

Поведение GC при штатных «нечего делать» — без падений: нет ключа шифрования → `Warning` в лог и выход (это ошибка конфига); пустой список хранилищ → `Debug` в лог и выход (нормальная ситуация для свежей установки, чтобы не засирать логи на каждом тике).

## Очистка устаревших бэкапов

Дашборд знает срок хранения бэкапов для каждого пользователя по его тарифу. Агент периодически забирает у дашборда пачку бэкапов, у которых срок истёк, и удаляет их объекты в хранилище. Затем сборщик чанков (раздел выше) подбирает осиротевшие куски — связка двух сервисов даёт полную очистку.

1. По таймеру (по умолчанию раз в 6 часов) запрашивает у дашборда пачку (до 50) просроченных записей. Дашборд считает срок по тарифу владельца агента.
2. Для каждой записи резолвит хранилище по имени из конфига `Storages[]` и удаляет `dump.*.enc` и (если был файловый бэкап) манифест по `ManifestKey` из записи — один и тот же код работает и для нового `manifest.json.gz.enc`, и для легаси `manifest.json.enc`.
3. После успешного удаления объектов шлёт `DELETE` записи дашборду — она исчезает из истории бэкапов в UI.
4. Если хранилище из записи больше нет в `Storages[]` (его вывели из эксплуатации), агент одним батчем сообщает дашборду «storage не найден» — такие записи в UI получают плашку «Очистка невозможна» и две кнопки: «Попробовать снова» (после возврата хранилища в конфиг) и «Удалить запись» (без удаления объектов в хранилище — пользователь осознанно соглашается на утечку).

Особенности:

- **Работает на всех провайдерах** (S3, Azure Blob, LocalFs, SFTP, WebDAV). В отличие от `ChunkGcWorker`, retention для удаления использует только `IUploadProvider.DeleteAsync`, а он реализован у каждого провайдера. На SFTP/WebDAV чанков нет — удаляется только дамп. На S3/Azure Blob/LocalFs дамп и манифест удаляются по ключу, осиротевшие чанки уберёт следующий тик `ChunkGcWorker`.
- **Первый запуск** — через 15 минут после старта агента.
- **Не пересекается с backup/restore.** На время sweep берёт общий lock с меткой `retention`.
- **Per-record изоляция.** Ошибка удаления одного объекта не валит остальные записи в пачке — они обработаются на следующем тике.

## Настройка `RetentionSettings`

```json
"RetentionSettings": {
  "Enabled": true,
  "IntervalHours": 6,
  "BatchSize": 50
}
```

Всё опционально — блок в шаблон `appsettings.json` не попадает, по умолчанию retention включён с параметрами 6/50.

- **`Enabled`** — выключает worker целиком. `false` если хотите чистить вручную или делать это внешним инструментом.
- **`IntervalHours`** — как часто запрашивать у дашборда пачку просроченных. Минимум 1 час. Чаще — место освобождается быстрее, но больше HTTP- и S3-вызовов.
- **`BatchSize`** — сколько записей забирать за один тик. Слишком большой батч долго удерживает общий lock и блокирует backup/restore. Для медленных S3-эндпоинтов уменьшайте.
